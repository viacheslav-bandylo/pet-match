**Тестовое задание**

1. Время выполнения: 2–4 часа (ориентировачно)  
2. После сдачи будет короткий созвон (10–15 минут): вы объясните ключевые решения и trade-offs  
3. Нам важнее качество решений и архитектура, чем «идеальная полнота»

**Описание проблемы:**

Люди часто ошибаются при выборе питомца не потому, что им всё равно, а потому что они **недооценивают долгосрочные последствия**: бюджет, время, условия жизни, риски для семьи (дети), стресс и т.д.

Мы хотим сервис, который:

4. собирает вводные о пользователе/семье/условиях  
5. оценивает совместимость с выбранным типом питомца  
6. может сказать: **«Это плохая идея для вас»**  
7. объясняет **почему**  
8. предлагает **альтернативы**, которые лучше подходят

**Задача:**

Сделать небольшой backend-сервис (REST API) для оценки совместимости с питомцем. Логика правил должна быть **конфигурируемой**, т.к. в будущем мы хотим поддерживать **50+ типов питомцев** и иметь возможность **менять правила без изменения кода**.  
Также правила должны быть изложены так, чтобы потенциально их мог корректировать не-инженер (например, через файл конфигурации).

**Требования:**

**1\) API: POST /evaluate**

Запрос (пример структуры — вы можете расширить, но не усложняйте чрезмерно):

{  
  "pet\_type": "dog",  
  "profile": {  
    "apartment\_size\_m2": 40,  
    "has\_children": true,  
    "monthly\_budget\_rub": 20000,  
    "work\_hours\_per\_day": 9  
  }  
}

Ответ:

{  
  "compatible": false,  
  "risk\_level": "high",  
  "reasons": \[  
    "…человеко-понятные причины…"  
  \],  
  "alternatives": \[  
    {"pet\_type": "cat", "why": "…"},  
    {"pet\_type": "hamster", "why": "…"}  
  \]  
}

**Важно:**

* **reasons** должны быть понятными человеку (не просто “rule\_17 failed”)  
* **alternatives** должны быть логически связаны с исходными вводными (не рандом, на ваше усмотрение)

**2\) Конфигурируемые правила (обязательно)**

Вместо “зашитых” правил в коде, сделайте правила внешними:

* файл rules.yaml или rules.json (на ваш выбор)  
* минимум **3 типа питомцев** в конфиге (например: dog/cat/fish) — как демонстрация расширяемости.   
* возможность добавления нового типа питомца **без изменения кода** (только через конфиг)

Правила могут быть простыми (условия/пороговые значения), но должны поддерживать:

* несколько условий  
* разные уровни риска  
* тексты объяснений

Примерно в таком духе (но форму выбираете вы):

* ограничения по жилью  
* ограничения по бюджету  
* ограничения по времени  
* семейные ограничения (дети)  
* общий “скоринг” или система блокирующих правил

Вы сами решаете:

* делаете ли вы scoring  
* делаете ли вы “hard reject” vs “high risk”  
* какие пороги разумны  
  Главное — чтобы логика была последовательной и объяснимой.

**3\) Endpoint для загрузки/перезагрузки правил (обязательно)**

Сделайте **один** из вариантов:

**Вариант A (предпочтительный):**

* POST /rules/reload — перечитывает конфиг с диска (без рестарта сервиса)

**Вариант B:**

* POST /rules — принимает JSON/YAML и обновляет rules in-memory

Смысл: показать, что правила **живут отдельно от кода**.

**4\) Тесты (обязательно)**

Минимум:

* **5 тестов** на pytest

Должны быть тесты:

* на “compatible” кейс  
* на “reject/high risk” кейс  
* на альтернативы  
* на reload rules (или update rules)  
* на edge-case (например, отсутствует ключ в profile)

**5\) Качество инженерии**

Обязательно:

* Pydantic-модели  
* Используйте type hints (typing) там, где это уместно  
* понятная структура проекта  
* обработка ошибок с адекватными HTTP статусами  
* README с запуском  
* Docker — опционально (но будет плюсом, тем более на уровне Senior).
